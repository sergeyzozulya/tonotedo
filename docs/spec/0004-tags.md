# Tags

## Problem

Groups are hierarchical and exclusive: an entry lives in exactly one group. Real classification is cross-cutting — a meeting note can be "work" and "interview-prep" and "Q2-planning" all at once, and none of those is its "home." Tags fill that gap.

The tension to resolve: a single global tag list grows into a mess as the user accumulates years of entries; per-folder scoping helps containment but, taken too far, fragments tags that should be shared. The model needs both, with a clear rule for which wins. Tags also carry their own light metadata (description, color, icon) so the user can curate the taxonomy without it turning into another database.

## User stories

- I tag a note `#followup`. Another note picks the tag from autocomplete. Both show up in the `#followup` view.
- Inside my "Project Atlas" group, I want `#decided` and `#parked` tags. They make no sense outside this project. I scope them to the group.
- I give `#followup` a description ("things I'll come back to within a week") and an amber color. Hovering the tag anywhere shows the description; entries with it carry an amber chip.
- I rename `#followup` to `#follow-up`. All entries that used the old form update.
- I delete a tag. The tag disappears from autocomplete, but entries that referenced it keep the string (so I can recover) and the tag re-appears as "unmanaged" until I confirm removal.
- I want `#project/atlas` and `#project/borealis` to live under a `#project` parent in the sidebar.

## Behavior

**Form.** A tag is a short string. Allowed characters: letters, digits, `-`, `_`, `/`. Slashes denote hierarchy in the UI (`project/atlas` is a child of `project`). Tags are case-preserving but case-insensitive for matching.

**Storage on entries.** Tags live in two coexisting surfaces, neither of which rewrites the other:

- Frontmatter `tags:` — an array of strings:

  ```yaml
  tags: [followup, project/atlas]
  ```

- Body `#tag` — written inline as part of the markdown text. Renders as a styled tag chip in the editor (see 0006). The literal `#tag` text is preserved on disk.

Both surfaces are first-class and both are indexed. An entry is "tagged with X" if X appears in frontmatter, in the body, or in both. The user picks the surface that fits how they think: structured metadata at the top (`tags:`), or tags woven into prose (`as discussed in #standup`). Removing a tag from one surface does not remove it from the other; the property panel and the body edit are independent gestures.

Search and the tag browser treat the union: a query for `#followup` finds entries that have it anywhere.

**Metadata.** Tags can carry light metadata:

- `name` — string identity (required)
- `description` — markdown text shown in the tag browser and on hover
- `color` — visual hint. Named tokens are the preferred form; the v1 palette is eight tokens: `slate`, `red`, `amber`, `green`, `teal`, `blue`, `violet`, `pink`. Each token resolves to a light-mode and dark-mode value via the active theme. Hex strings (`#ffb800`) are accepted as an escape hatch; they render verbatim and do not theme-adapt.
- `icon` — single emoji or icon name

Metadata is optional. A tag used in an entry without any declaration is still valid; it just has empty metadata.

**Storage of metadata.**

- Global tag metadata lives in a library-root `_tags.md`, in frontmatter:

  ```yaml
  ---
  tags:
    - name: followup
      description: Things I need to come back to.
      color: amber
      icon: ⏳
    - name: project/atlas
      color: blue
  ---
  ```

  The body of `_tags.md` is freeform user notes about the taxonomy.

- Scoped tag metadata lives in a group's `_group.md` under `scoped_tags`, with the same shape.

**Scope.**

- **Global tags** are visible from anywhere, suggested everywhere. Declared (with metadata) in `_tags.md`, or used implicitly without declaration.
- **Scoped tags** are declared in a group's `_group.md` under `scoped_tags` with the same metadata shape as global tags. They are suggested only inside that group and its descendants — scope flows down the tree, never sideways. Siblings of the declaring group do not see the tag; to share a tag across siblings, declare it on the common parent (or globally). On an entry inside the declaring group, both global and the scope's tags are offered.
- An entry can carry any tag string in its frontmatter regardless of scope; scoping affects suggestions and the tag browser, not validity. Moving an entry out of a group does not strip its scoped tags from frontmatter — the user can prune them if they want.

**Hierarchy.** `parent/child` is purely a display convention. The tag string is the identity; renaming `parent` does not rename `parent/child` automatically (that would be implicit magic) but the rename UI may offer "also rename children" as one action. Metadata is per exact tag string: `#project` having a description does not imply `#project/atlas` inherits it.

**Operations.**

- Add tag to entry → frontmatter array update; index update.
- Edit tag metadata → rewrite `_tags.md` (or the relevant `_group.md`); no entry files change.
- Rename tag → walk every entry that references it; rewrite. Confirmed batch operation. If the tag had metadata, the metadata entry is renamed in place.
- Delete tag → remove from suggestions and from `_tags.md`; entries still carry the string until the user runs a "clean up" gesture or removes manually.
- Merge tag → "merge A into B" rewrites all entries and removes A's metadata.

**Autocomplete.** When typing `#` in the body or selecting tags in the property panel: scoped tags of the current entry's group first, then global tags, ranked by recent use. Declared tags (those with metadata) rank above used-but-undeclared ones.

## Non-goals

- No automatic tagging by content (no "AI tag suggestions" in v1; this would violate the no-AI-everywhere anti-pillar unless explicit).
- No tag aliases (`#wip` ≡ `#in-progress`). User can merge instead.
- No required tags or tag validation rules.
- No per-entry override of tag metadata (an entry cannot color one of its tags differently from the global declaration).

## Edge cases

- **Body and frontmatter differ.** Not a conflict. Body has `#a #b`, frontmatter has `[a, c]` → the entry is tagged with `{a, b, c}`. Neither surface is rewritten. Removing `b` requires editing the body; removing `c` requires editing the property panel.
- **Tag string with disallowed characters.** Imported entries may carry `#foo!`. Stored verbatim, flagged as "non-canonical" in the tag browser. App-created tags are restricted.
- **Case difference.** Saving `#Followup` when `#followup` exists: normalize to the existing canonical form on autocomplete; preserve user's casing if they explicitly type a new variant.
- **Scoped tag string colliding with a global tag.** Global wins for the canonical definition; the scoped declaration is a no-op with a warning.
- **Metadata for a never-used tag.** Allowed — the user may pre-declare a tag in `_tags.md` before using it. It appears in autocomplete.
- **Renaming a tag with thousands of references.** Long operation; show progress; the rename is journaled so a crash mid-rename leaves a recoverable state.
- **Restore from trash after a rename.** Trashed entries are not rewritten by renames, so a restored entry may carry the old tag string. It reappears in the tag browser as unmanaged — same recovery surface as a deleted tag.

## Acceptance criteria

- Adding `#foo` in the body makes the entry searchable by `foo` without touching frontmatter.
- Adding `foo` to frontmatter `tags:` makes the entry searchable by `foo` without touching the body.
- A scoped tag in group X is suggested inside X but not outside.
- A tag with a defined color shows that color consistently in the sidebar, the tag browser, and the chip on entries.
- A tag with a description shows the description on hover in the tag browser and on entries.
- Renaming a tag rewrites all entries that reference it; deleted entries in trash are not rewritten.
- Removing a tag from suggestions does not delete it from existing entries.
- `parent/child` tag renders nested in the tag browser.
